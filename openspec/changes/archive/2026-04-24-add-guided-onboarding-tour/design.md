## Context

FileFlash-Bridge 当前的核心体验跨越两个界面：移动端工作台负责启动服务、展示地址/二维码、查看接收内容与管理共享文件；浏览器门户页负责上传文件、发送文本和下载共享文件。近期移动端工作台在朝“首屏更紧凑、说明文字更少”的方向收敛，这有利于提高熟练用户效率，但也会放大首次使用者的理解成本。

现状上，移动端主要由 [App.tsx](d:\company\FileFlash-Bridge\App.tsx) 和 [src/app/useAppModel.ts](d:\company\FileFlash-Bridge\src\app\useAppModel.ts) 驱动。App 端已经存在本地状态能力：通过现有存储网关持久化会话数据，因此本次改动不需要引入服务端账户体系或新的网络接口。

约束包括：
- 必须优先复用 React Native 共享代码，保证 iOS 与 Android 行为一致。
- 不得为了新手引导改变现有局域网传输协议、权限路径、安全模式或项目/文件数据模型。
- 引导必须可跳过，且不应要求用户真实执行某个动作后才能进入下一步，否则会把“教程”变成“强制流程”。
- 引导文案要帮助理解现有能力，但不能重新把大量静态说明塞回首屏。

## Goals / Non-Goals

**Goals:**
- 为移动端工作台提供首次使用可自动弹出的分步式新手引导，并支持上一步、下一步、跳过、完成。
- 在移动端本地持久化“已完成/已跳过”的状态，并用版本号控制未来在重大流程调整时重新触发引导。
- 让引导基于现有页面中的真实控件和区域进行讲解，而不是跳转到脱离上下文的独立教程页。
- 保持 iOS 与 Android 的基本体验一致，同时允许少量平台层交互差异。

**Non-Goals:**
- 不实现跨设备同步的新手引导进度；引导状态只记录在 App 本地。
- 不新增服务端 API、用户账户、分析埋点或远程配置能力。
- 不把新手引导做成必须逐项完成的强约束 checklist，也不自动触发服务启动、权限申请、文件导入或文本提交。
- 不顺带重做现有传输流程、权限文案或信息架构；本次只补“如何使用”的逐步引导层。

## Decisions

### 1. 引导聚焦移动端工作台：只为 App 内的核心控件与流程提供分步讲解

本次新手引导仅覆盖移动端工作台，不为浏览器门户页新增分步式新手引导层。原因是浏览器端体验当前更偏“快速投递”，且后续可与门户页的信息架构/空态说明等改动一并评估是否需要额外教程；在本次变更里聚焦 App 工作台能更快降低首次使用者在“启动服务与获取地址”阶段的理解成本。

### 2. 采用锚定现有页面元素的 coach-mark/spotlight 覆层，而不是独立教程页或轮播页

移动端应在当前页面内直接解释“这里是什么、下一步做什么”。这意味着引导层要能高亮当前页面上的服务区、地址区、共享区、当前项目区等真实区域，并在其附近展示简短说明和步骤按钮。

相比独立教程页：
- 锚定式引导不会让用户失去当前上下文。
- 紧凑布局仍然可以保留，因为说明只在首次或手动打开时出现。
- 用户可以在看到提示时立刻理解真实控件的位置。

独立轮播页虽然实现更简单，但会增加“看完教程后还要重新回忆控件在哪”的认知成本，因此不采用。

### 3. App 端使用现有本地存储边界扩展一个轻量 UI 元数据文件

移动端已经通过 `InboundStorageGateway` 持久化会话数据，因此新手引导状态不应再单独引入新的原生依赖（如额外加一个 AsyncStorage 库）。本次设计选择在现有本地存储边界中补一个轻量 UI 元数据文件，记录：
- 当前引导版本号
- 工作台引导是否完成
- 工作台引导是否跳过
- 最近一次手动重新打开的时间戳（若后续需要）

这样可以继续复用现有 React Native 共享代码与文件系统适配层，不新增 Native Module。

备选方案是把引导状态直接塞进现有 `StorageSnapshot`。该方案虽然也能工作，但会把“会话业务数据”和“界面提示偏好”混在一起，后续维护边界不清，因此不选。

### 4. 步骤定义采用声明式配置，并支持目标缺失时的降级锚点

两端的步骤都应该定义为配置数组，而不是写成大量 if/else 的命令式逻辑。每个步骤至少包含：
- `id`
- `title`
- `body`
- `target`
- `placement`
- `surface`
- 可选的 `variant` 或 `fallbackTarget`

移动端目标通过稳定的 ref / layout measurement 获取。若某个精确目标因为当前状态不存在（例如服务未启动时没有真实地址文本），则降级锚定到所属面板容器，并切换为对应状态的说明文案。

这样可以让同一套 tour 在“服务已启动”和“服务未启动”两种情况下都能成立，不要求用户先做出某个动作。

### 5. 引导只做说明，不替用户触发业务动作；跳过与关闭始终可见

本次引导的定位是“解释页面和流程”，不是“强制完成配置向导”。因此：
- 引导不会自动启动服务、切换安全模式、上传文件或提交文本。
- 任何一步都必须能通过“跳过”或关闭退出。
- Android 上引导展示期间优先拦截硬件返回键，用于关闭引导或回退步骤，而不是直接退出当前页面。
- iOS 上不新增系统级交互，只通过页面内显式按钮退出，并保证安全区内展示完整控件。

备选方案是做成必须真实完成某一步后才能解锁下一步的 onboarding checklist。该方式更重，也会让局域网环境、权限状态、是否已有共享文件等前置条件变成阻塞项，因此不采用。

### 6. 手动重新打开入口优先放在“当前页面语境内”，而不是隐藏到深层设置页

用户在忘记流程时，应当能在当前页面附近重新查看引导，而不是去系统设置或二级页面里找帮助。设计上：
- App 端在工作台顶部工具区或显式帮助入口提供“查看引导”。

这样做比放进隐藏设置页更符合“用时即取”的帮助模型，也更适合首次使用者的心智。

## Risks / Trade-offs

- [高亮目标在某些状态下不存在或不在首屏] → 只为稳定、首屏可见的核心区域建立步骤；目标缺失时回退到所属面板或容器级锚点。
- [紧凑页面叠加引导后显得拥挤] → 引导文案控制为单段短句；每次只高亮一个区域；在窄屏下使用纵向布局和较大的底部安全留白。
- [引导状态一旦持久化，后续流程变化后用户可能看不到新教程] → 使用版本化存储 key；当关键流程调整时只需升级版本号即可重新触发。
- [Android 返回键与 iOS 安全区处理不一致] → 共享同一套引导状态机，但单独为 Android 补返回键拦截，为 iOS 补安全区边距校准。
- [帮助层干扰正常操作] → 跳过/关闭始终可见；完成或跳过后不再自动弹出；手动入口保留但不常驻大面积横幅。
- [把 UI 元数据接到现有存储边界后，边界职责变宽] → 单独存放轻量 UI metadata 文件，而不是污染核心会话快照；只暴露最小读写接口。

## Migration Plan

1. 在移动端现有本地存储层新增 UI 元数据读写能力，默认视为“未完成引导”。
2. 在工作台渲染树中接入引导层、目标 ref 和手动重新打开入口，但不修改现有服务/上传/共享业务逻辑。
3. 现有已安装用户在升级后，由于本地尚无新手引导状态，会自动看到一次新版教程；跳过或完成后不再自动弹出。
4. 若需要回滚，只需移除引导入口和自动触发逻辑；残留的本地 tutorial metadata 不会影响原有传输功能。
