## Context

当前工作台已经有项目历史区域，但布局和交互还停留在自定义卡片阶段。本次需求明确改为“项目历史侧边栏”，并且开始使用 `react-native-paper` 的 Drawer 组件体系。相比顶部入口 + 覆盖层，侧边栏更符合“频繁切换项目”的使用方式，也更容易和主工作台形成稳定的双栏结构。

现有项目创建、切换和删除能力已经在 [useAppModel.ts](d:\company\FileFlash-Bridge\src\app\useAppModel.ts) 与 [inboundStorageGateway.ts](d:\company\FileFlash-Bridge\src\modules\file-access\inboundStorageGateway.ts) 中实现，因此本次改动仍然聚焦在共享 UI 层，不改动底层持久化语义。

## Goals / Non-Goals

**Goals:**
- 在主工作台中提供稳定的项目历史侧边栏。
- 引入 `react-native-paper` 并使用 Drawer 组件承载项目列表与活跃态。
- 保留新建、切换、删除项目动作，并继续复用现有删除确认提示。
- 将“新建”收敛到侧边栏标题栏右上角，并将删除操作收敛到每个项目行尾部的 `...` 菜单。
- 让“启动服务”和“选文件”留在主页工作区，而不是塞进项目历史侧边栏。
- 在宽屏下提供左侧常驻边栏，在窄屏下提供可从左侧打开的 Drawer 式边栏，并确保其不覆盖或改变状态栏。
- 保持浏览器文本/文件入站、共享文件下载和分块重试行为不回归。

**Non-Goals:**
- 不修改项目、消息、文件的持久化结构。
- 不引入新的导航库或页面路由。
- 不改变浏览器端协议、删除确认文案或权限流程。
- 本轮不强制引入额外图标原生依赖；Drawer 可先使用文本项满足导航需求。

## Decisions

### Decision 1: 使用 React Native Paper Drawer 承载项目历史导航

选择：
- 在 App 根部接入 `PaperProvider`。
- 在侧边栏中使用 `Drawer.Section` 承载项目历史列表。
- 项目历史项改为更接近导航列表的行式布局，展示项目名称、创建日期、文件数和消息数，而不是卡片块。

原因：
- 这能让项目历史从“临时卡片列表”过渡到更稳定的导航容器。
- 后续如果需要增加更多工作台导航项，也可以继续沿用同一组件体系。

### Decision 2: 维持侧边栏语义，并在窄屏下使用真正的左侧抽屉

选择：
- 宽屏下保持左侧边栏。
- 窄屏下复用相同 Drawer 内容，但以真正的左侧抽屉覆盖在主内容之上，并通过一个菜单图标按钮打开；抽屉本身保持在安全区域内，不覆盖状态栏。
- 新建和切换项目只更新活跃项目，不再引入“关闭历史层并回到顶部”的额外 UI 状态，也不在抽屉内部提供单独的关闭按钮。

原因：
- 这更符合“侧边栏”需求，也更贴近当前页面已有的双栏/堆叠响应式布局。
- 这样在手机竖屏上仍然保留“侧边栏”的空间语义，而不是退化成顶部信息块。

### Decision 3: 项目动作继续复用现有模型接口

选择：
- 继续使用 `createProject`、`selectProject`、`deleteProject`。
- 删除操作仍走根级 `Alert` 确认。
- 共享文件列表中的“项目”跳转仍通过活跃项目切换完成。

原因：
- 现有模型已经具备所需能力，没有必要为了 Drawer 导航重做数据层。
- 这样可以把风险限制在界面重构范围内。

### Decision 4: 用标题栏动作 + 行内菜单承载最小项目管理能力

选择：
- Drawer 主区负责展示项目列表与切换。
- 侧边栏标题栏只保留标题与右上角“新建”入口。
- 不再提供全局“删除当前项目”按钮。
- 每个项目行尾部提供 `...` 菜单，并在菜单中承载删除动作。
- “启动服务”和“选文件”保留在主页工作区卡片中，不放入项目历史侧边栏。
- 选中文件所属项目时，工作台摘要、消息列表和文件列表都继续跟随 `activeProjectId` 自动联动。

原因：
- Drawer 更适合做“导航”，标题栏与行内菜单能把管理动作压缩到最小必要范围。
- 删除动作绑定到具体项目行后，交互语义更清晰，也能去掉额外的全局破坏性按钮。

## Risks / Trade-offs

- [引入新 UI 依赖带来接入成本] -> 将接入范围收敛到 `react-native-paper` 与 `PaperProvider`，优先使用无需额外图标原生依赖的 Drawer 形态。
- [窄屏下侧栏可能遮挡主内容] -> 使用左侧抽屉 + 遮罩关闭，避免把项目历史直接挤进主滚动区域。
- [行内 `...` 菜单让删除多一步点击] -> 用菜单聚合破坏性操作，换取更干净的列表结构与更低的误触风险。
- [当前代码中已有进行中的 UI 重构] -> 以本次需求为准重新收敛实现，避免继续沿着覆盖层方案扩展。

## Migration Plan

1. 在 [App.tsx](d:\company\FileFlash-Bridge\App.tsx) 中移除覆盖层方向的 UI 状态，恢复/整理成侧边栏布局。
2. 安装并接入 `react-native-paper`，为应用根部增加 `PaperProvider`。
3. 用 Drawer 组件替换项目历史列表区域，并补上标题栏右上角新建入口与项目行 `...` 菜单。
4. 验证切换项目后，顶部摘要、消息列表、文件列表和共享文件跳转都跟随新的活跃项目。
5. 补充共享代码测试，覆盖 Drawer 渲染、项目切换和删除确认。
