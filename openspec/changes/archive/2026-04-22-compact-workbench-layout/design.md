## Context

当前工作台首页在 [App.tsx](d:\company\FileFlash-Bridge\App.tsx) 中由多个独立大块组成：标题区右侧有两个文字按钮，标题下方有三张状态卡片，服务区又单独展示标题、网络状态、地址、启动按钮、二维码或空态。这种排布在视觉上强调了“状态容器”和“操作按钮”，但没有优先保证用户最关心的三类信息同时进入首屏：

- 服务现在是否可用
- 当前地址是否可复制、是否需要刷新
- 当前有哪些共享文件正在等待浏览器下载

现状还存在几类重复表达：

- “服务离线 / 未启动”在页头状态、服务卡片、地址区和空态中多次出现
- “复制链接 / 刷新地址”与地址本身分离，用户需要在两个区域之间建立心智对应
- “连接 / 共享 / 模式”三个状态采用相同的大卡片承载，但实际内容极短

这次设计只针对 React Native 共享层的工作台首页布局，不改变服务控制器、网络解析、二维码生成、安全模式切换、文件共享、文件导出和浏览器门户接口。iOS 与 Android 继续共享同一套页面结构与组件，仅在触控反馈、安全区和系统视觉细节上遵循各自平台默认表现。

## Goals / Non-Goals

**Goals:**

- 让服务状态、地址、地址操作、共享文件列表在常见手机首屏中拥有更高的信息密度和更清晰的主次关系。
- 将顶部状态看板压缩为轻量摘要条，减少大卡片造成的垂直浪费。
- 把“复制链接”“刷新地址”移动到地址展示附近，并根据服务是否可用决定显隐或禁用状态。
- 将服务区整合为单个核心卡片，减少“未启动”状态和服务描述的重复出现。
- 在不改变业务语义的前提下，为共享文件列表腾出额外高度，确保首屏能直接看到更多共享项。
- 保持 iOS 与 Android 采用 React Native 共享实现，不新增原生依赖或平台专属业务逻辑。

**Non-Goals:**

- 不修改服务启动、停止、地址刷新、二维码生成和安全 `key` 轮换逻辑。
- 不改变浏览器上传、文本提交、共享文件下载、文件导出和项目历史 Drawer 的行为语义。
- 不引入新的导航结构、设置页、原生模块或额外 UI 框架。
- 不重写服务状态来源、网络诊断算法或后台保活机制。

## Decisions

### Decision 1: 用紧凑状态摘要条替代大尺寸状态卡片

选择：

- 将“连接 / 共享 / 模式”改为标题下的一行紧凑状态摘要项，每项由图标、短标签和值组成。
- 摘要项继续直接消费现有模型数据，例如活跃连接数、共享文件数和当前安全模式，不新增状态源。
- 当屏幕过窄时允许自动换行成两行，但仍保持“轻量状态项”而非回退到整块卡片。

原因：

- 当前三个指标信息量小、更新频率高，更适合做成 glanceable summary，而不是强调边框与背景的大卡片。
- 该调整可以直接释放约一整块卡片行高，为下方服务区和共享文件区让出空间。

备选方案：

- 保留卡片但缩小高度。未选，因为仍然保留了冗余边框和容器层级，节省空间有限。
- 将三个指标移入服务卡片内部。未选，因为会混淆“全局状态摘要”和“服务控制区”的信息边界。

### Decision 2: 将地址展示与链接操作合并为同一行上下文操作

选择：

- 地址成为服务卡片中的核心信息对象，右侧内联放置复制和刷新图标按钮。
- 当服务未启动或尚未获得可达地址时，隐藏复制与刷新操作，只保留简短状态文案或单行占位。
- 地址展示优先体现“当前访问入口”，而不是把按钮提升为与服务主按钮同级的一级操作。

原因：

- 用户只会在看到有效地址时产生复制或刷新的需求，把操作放在地址附近更符合对象关联。
- 隐藏无效操作能降低视觉噪声，也减少用户点击后才得知“当前不可用”的挫败感。

备选方案：

- 保留顶部两个文字按钮，只在未启动时禁用。未选，因为禁用后的大按钮仍会持续占用首屏注意力。
- 将复制和刷新合并进溢出菜单。未选，因为这两个操作在服务运行时仍然需要足够直接，不应增加额外点击。

### Decision 3: 将服务状态、网络诊断与主操作合并为单个服务中心卡片

选择：

- 保留服务卡片作为首页第一优先级区域，但压缩为“状态标题 + 地址行 + 启动/停止按钮 + 条件化二维码/说明”结构。
- “服务在线 / 离线”通过标题旁状态点或次级文本表达，不再在多个区域重复渲染“未启动”。
- 网络诊断从整行横条降级为服务标题旁的紧凑状态项或可点开说明的图标提示；存在明确错误时仍展示高辨识度反馈。

原因：

- 服务区仍是首页最关键的操作入口，应该保留明确主按钮，但不需要重复展示同一服务状态。
- 网络诊断多数时间是背景信息，适合作为次级状态保留可发现性，而不是长期占用整行高度。

备选方案：

- 完全隐藏离线状态，只保留“启动服务”按钮。未选，因为用户仍需要快速理解当前为什么没有地址和二维码。
- 将网络诊断完全移入设置页。未选，因为当前网络是否可达会直接影响服务可用性，仍需在首页可见。

### Decision 4: 保留访问模式控制，但压缩为次级可读区域

选择：

- 安全模式相关控制继续保留在工作台首页，不移入独立设置页。
- 访问模式区域从当前整块卡片压缩为服务卡片下方的次级控制区或可折叠区域，默认展示当前模式和值得关注的短说明。
- 帮助说明继续可达，但改为图标说明或辅助文本，不额外占据整块高度。

原因：

- 访问模式直接影响链接可见形态和安全边界，属于服务控制上下文的一部分，不适合被完全隐藏。
- 该区域虽然重要，但交互频率低于启动/停止服务和复制地址，因此应弱化视觉权重。

备选方案：

- 原样保留独立卡片。未选，因为仍会与本次“压缩首屏”目标冲突。
- 完全迁移到设置。未选，因为会削弱用户对当前简单模式/安全模式的即时感知。

### Decision 5: 通过内容排序和条件渲染为共享文件列表腾出首屏空间

选择：

- 将共享文件区保持在服务区之后的高优先级位置，并减少顶部摘要与服务区的垂直占用。
- 服务未启动时，不展示无意义的地址操作区和多余说明，把高度留给共享文件列表。
- 共享文件区保持“标题 + 选文件入口 + 列表”的结构，但要求在普通手机首屏可见 3 至 4 个条目或明确空态。

原因：

- 共享文件列表是当前工作台最直接承载“待处理任务”的区域，提高其可见性比放大状态容器更能帮助用户完成工作。
- 通过条件渲染腾挪空间比硬压缩列表行高更稳妥，不会损害文件名称、项目归属或操作可点击性。

备选方案：

- 继续维持现有顺序，只缩小字体和边距。未选，因为收益有限，且会伤害可读性。

## Risks / Trade-offs

- [状态摘要过于轻量，用户可能忽略关键指标] → 通过图标、对比度和固定顺序保证可扫读性，同时在服务区保留更明确的服务状态。
- [隐藏无效操作后，用户可能不知道地址可刷新] → 在服务启动且地址出现后立即展示刷新图标，并在地址变化提示中继续说明旧链接失效。
- [网络诊断弱化后，用户可能错过“无可用局域网”原因] → 对不可达状态保留醒目的告警色和可点开的详细说明，而不是完全静默。
- [压缩访问模式区域可能削弱安全模式存在感] → 默认直接显示当前模式值，并保留帮助入口和切换控件，不改变模式切换路径。
- [共享列表前移后，服务卡片中的二维码可见面积下降] → 仅压缩外围留白，不减少二维码在运行态下的最小可扫尺寸。
- [iOS 与 Android 上的触控反馈表现不一致] → 继续采用 React Native 共享实现，分别复用系统默认的 ripple / highlight 与安全区处理，不引入平台分叉布局。

## Migration Plan

1. 先在共享 React Native 页面层重排工作台结构，抽出新的状态摘要行与服务地址行，不改动 `useAppModel` 的数据来源。
2. 重构服务卡片中的地址与操作区，使复制、刷新与地址文案共处一行，并增加按服务状态条件渲染的逻辑。
3. 压缩访问模式和网络诊断区域的视觉层级，确保异常状态仍有可发现的反馈。
4. 重新分配共享文件区的可用高度，并校验空态、少量文件和多文件场景的首屏可见性。
5. 在 iOS 与 Android 真机或模拟器上分别验证安全区、按钮热区、服务运行态二维码、无网络态提示和长地址截断表现。
6. 如需回滚，可恢复 [App.tsx](d:\company\FileFlash-Bridge\App.tsx) 中现有的状态卡片与服务卡片布局；本次不涉及数据结构变化，因此无需数据迁移。

## Open Questions

- 地址行在超长 URL 场景下采用单行截断还是两行展开，需结合最终图标按钮宽度确认。
- 访问模式区域最终采用默认展开的紧凑行，还是可折叠的“高级选项”，可在实现阶段按首屏实际高度再决定。
