## Context

当前移动端工作台已经接入了部分 `react-native-paper` 能力，但主体界面仍以 `App.tsx` 中的大量自定义基础控件和手写样式为主，主题色也继续沿用了偏浅黄、偏纸面的旧视觉方案。[theme.ts](d:\company\FileFlash-Bridge\src\app\theme.ts) 中的 `background`、`surface`、`surfaceMuted`、`secondary` 等颜色会直接影响主页卡片、项目历史、消息列表、文件列表和状态提示；同时，浏览器门户页 [portalDocument.ts](d:\company\FileFlash-Bridge\src\modules\portal\portalDocument.ts) 也仍然使用同样偏黄的背景、卡片和强调色。因此这次调整会是一个同时覆盖 React Native App 与浏览器门户页的跨主题、跨页面区块的共享 UI 改造。

本次变更的直接目标是让 React Native App 主工作台全面迁移到以 React Native Paper 为核心的共享组件体系，并让浏览器门户页同步升级为同一视觉系统下的网页界面。这里的“Apple 风格”指更克制的配色、更轻盈的分层、更统一的圆角与间距，以及更强调信息秩序的界面结构，而不是追求平台专属原生私有能力。需要注意的是，浏览器门户页不会直接使用 React Native Paper，而是通过重构其 HTML/CSS token、布局和反馈组件来对齐同一风格方向。文件上传、文本提交、下载分块重试、导出保存、权限申请与后台保活等核心能力不改变行为语义，只升级 App 与网页两端的承载界面和反馈样式。

约束：
- 优先复用 React Native 共享代码，避免为 iOS/Android 分别维护两套工作台 UI。
- 不新增自定义 Native Module，不改变现有文件访问、局域网服务、权限和存储边界。
- 所有与文件上传、文件保存、文本提交相关的状态反馈必须继续可见，不能因为界面重构而弱化错误提示、重试结果和危险操作确认。
- 浏览器门户页仍由本地服务直接输出 HTML/CSS/JS，因此其视觉升级必须在不引入复杂前端构建链的前提下完成。

## Goals / Non-Goals

**Goals:**
- 让主工作台中的按钮、菜单、列表、卡面、横幅、输入和操作反馈统一落到 React Native Paper 组件体系或基于其主题能力的共享封装。
- 用新的中性冷白/微灰主题替换现有淡黄色底色，建立更接近 Apple 质感的界面层次、圆角、阴影和间距系统。
- 重构主页服务区、项目历史侧边栏、消息列表、文件列表、共享文件列表与提示区，使其在信息架构上更统一。
- 重构浏览器门户页的英雄区、上传区、文本区、共享下载区和状态反馈，让网页与 App 在视觉语言上保持一致。
- 保持服务启动/停止、文件导入、文件共享、文件导出、文本复制、项目切换与删除确认等功能语义不变。
- 明确 iOS 与 Android 在状态栏、安全区、触控反馈、权限提示和系统外观上的差异处理方式。

**Non-Goals:**
- 不改变文件上传、文本提交、下载分块、项目存储、导出保存或删除语义。
- 不引入新的导航框架、页面路由或额外的原生模糊/玻璃拟态依赖。
- 不重写底层服务控制器、文件网关或权限实现。
- 不把浏览器门户页改造成新的前端工程或引入 React/Web UI 框架。

## Decisions

### Decision 1: 采用 Paper-first 的共享组件策略

选择：
- 在 App 工作台中，把按钮、图标按钮、菜单、列表项、表面容器、横幅、Chip、加载态和输入框优先替换为 `react-native-paper` 组件。
- 对确实需要项目语义的部件，保留少量共享包装层，但包装层必须建立在 Paper 组件与同一套主题 token 之上，而不是继续扩散手写 View/Text/Pressable 原子控件。
- 继续通过根级 `PaperProvider` 提供统一主题映射。
- 浏览器门户页不直接接入 Paper，而是复用同一套颜色、层级和状态语义，在 `portalDocument.ts` 中输出等价的 HTML/CSS 组件风格。

原因：
- 这能减少当前 `App.tsx` 中分散的自定义按钮/容器样式，降低后续 UI 维护成本。
- Paper 已经在项目中落地，无需再引入额外 UI 框架或原生桥接。

备选方案：
- 延续当前自定义控件体系，仅调整颜色和样式。未选，因为只能做表面翻新，不能真正解决组件行为与视觉规则分叉的问题。
- 全量引入新的导航/设计系统。未选，因为超出当前 change 范围，也会放大跨平台验证成本。

### Decision 1.5: 让 App 与浏览器门户共享同一视觉系统，而不是分别升级

选择：
- App 端和门户页统一使用同一视觉方向：中性冷白基底、低噪声边框、较大圆角、克制强调色和清晰状态反馈。
- 浏览器门户页延续当前静态 HTML 文档模式，但重建其 CSS 变量、hero/panel/card/button/banner 等样式结构。

原因：
- 用户要求“同时启动的网页也是这种风格”，因此不能只升级 App 端。
- 文件上传、文本提交和下载行为本身发生在门户页，若网页继续保留旧黄底，会破坏整体品牌与体验的一致性。

备选方案：
- 仅升级 App，门户页后续再说。未选，因为与用户最新要求冲突。

### Decision 2: 建立“中性冷白 + 克制强调色”的 Apple 风格主题，而不是继续沿用浅黄色底色

选择：
- 在 [theme.ts](d:\company\FileFlash-Bridge\src\app\theme.ts) 中重建配色 token，将 `background`、`surface`、`surfaceMuted`、`secondary` 等从浅黄纸面系调整为中性冷白、浅灰或轻微蓝灰色系。
- 保留少量强调色用于服务状态、主按钮和高亮态，但避免整页泛黄或高饱和暖色铺底。
- 用更清晰的分层规则表达 Apple 质感：高对比文本、轻量阴影、较大圆角、低噪声边框、局部半透明感，而不是厚重拟物。

原因：
- 用户明确要求移除“淡淡的黄色”，现有主题色正是主因。
- Apple 风格更依赖秩序、留白和细腻层次，而不是单一暖色氛围。

备选方案：
- 只把黄底换成纯白，不调整层级规则。未选，因为会让视觉变得单薄，不能真正升级质感。
- 强行加入原生毛玻璃/模糊效果。未选，因为需要新增依赖并提高平台差异成本，不是当前必需条件。

### Decision 3: 按信息架构分块重做工作台，而不是只替换控件外皮

选择：
- 将工作台拆成统一的几类 Paper 表面：全局状态区、服务控制区、项目历史导航区、消息区、文件区、共享文件区和提示反馈区。
- 项目历史继续保持左侧 Drawer/侧边栏语义，但行样式、菜单、头部操作和空态统一纳入新的 Paper 视觉体系。
- 服务启动/停止、文件导入和共享管理保留在主页工作区，但改为更一致的 Paper 操作布局和反馈层级。

原因：
- 仅替换控件而不重做信息区块，会让新旧视觉混杂，达不到“整体升级”的预期。
- 现有高频操作集中在少数几个区块中，适合按信息结构做一次性整理。

备选方案：
- 只改主题色和按钮。未选，因为消息列表、文件列表和项目历史仍会保留旧的布局与视觉噪声。

### Decision 4: 文件上传、文件保存、文本提交与重试逻辑保持不变，但 App 与门户页的反馈容器统一升级

选择：
- 浏览器侧文件上传、文本提交和共享文件下载分块重试逻辑不做行为变更。
- App 内与这些流程相关的反馈，例如上传后文件出现在哪个项目、导出保存入口、错误提示、空态、删除确认、共享状态切换，都改为统一的 Paper Banner / Menu / List / Dialog 风格。
- 门户页中的上传队列、文本提交反馈、共享下载状态、离线横幅和空态说明，也同步改为同一视觉规则下的网页样式。
- 对危险操作继续保留强确认，避免因为视觉轻量化而削弱风险提示。

原因：
- 该 change 的核心是视觉和组件体系升级，不应混入传输链路重构。
- 用户真正感知的是 App 端工作流的清晰度和质感，而不是底层网络协议。

备选方案：
- 顺手重做上传/下载交互协议。未选，因为会扩大风险并引入额外验收维度。

### Decision 5: 共享实现优先，单独记录 iOS 与 Android 差异

选择：
- 绝大多数界面与主题逻辑保留在 React Native 共享层中实现。
- iOS 上优先采用更轻的阴影、更多留白与 `dark-content` 状态栏风格，确保与系统视觉更协调。
- Android 上沿用 Paper elevation/ripple 反馈，但要控制阴影与底色，避免 Material 默认观感压过目标风格。
- 浏览器门户页保持纯 HTML/CSS/JS 输出，不新增前端框架，只在界面文案、Banner、button、panel 和状态样式上统一表现。
- 文件导出/保存、局域网权限、前后台保活等系统流程不改实现，只在界面文案、Banner、Snackbar 和按钮状态上统一表现。

原因：
- 这符合项目“默认优先共享代码”的原则，也能降低双端维护成本。
- iOS 与 Android 的差异更多体现在系统反馈与视觉细节，而不是功能布局本身。

## Risks / Trade-offs

- [全面替换为 Paper 会带来一次性重构成本] → 先抽出主题 token 和共享包装组件，再逐块替换页面区段，避免一次性推倒。
- [Apple 风格与 Paper 默认 Material 气质不完全一致] → 通过自定义 theme、圆角、颜色、阴影和组件组合来弱化默认 Material 味道，而不是直接使用默认皮肤。
- [门户页不是 Paper 组件，可能与 App 端出现风格偏差] → 先统一设计 token 和状态语义，再分别映射到 App 组件与门户页 CSS。
- [视觉升级可能削弱危险操作或异常提示的辨识度] → 为删除、失败、权限缺失、不可达网络等状态保留高对比警示色和明确文案。
- [窄屏 Drawer、状态栏、安全区在双端表现不同] → 在 iOS 和 Android 真机上分别验收侧边栏、状态栏、顶部间距和点击反馈。
- [上传、保存、文本提交流程虽然不改逻辑，但 UI 改动仍可能影响可发现性] → 针对文件导入、导出保存、文本列表、共享列表和门户页上传/下载/文本提交流程补充交互测试与手测。

## Migration Plan

1. 先重建共享主题 token，并同步映射到 Paper theme，替换现有淡黄色背景和表面色。
2. 抽取或重写少量共享 UI 包装层，使主按钮、次按钮、危险按钮、列表项、空态、横幅和菜单统一基于 Paper。
3. 分块改造工作台：项目历史侧边栏、主页服务区、消息/文件列表、共享文件列表、提示反馈区。
4. 重构 `portalDocument.ts` 中的门户页 CSS/HTML 结构，使网页与 App 视觉方向一致。
5. 更新单元测试/渲染测试，覆盖项目切换、服务控制、文件导入入口、菜单操作、门户页状态反馈和危险操作确认。
6. 在 iOS 与 Android 真机上验收状态栏、安全区、触控反馈、权限提示和深浅色外观兼容性，并执行浏览器门户页的桌面/移动端验收。
7. 如果上线后需要回滚，保留旧主题 token 与上一版工作台/门户页样式作为可恢复基线；本次不涉及数据迁移，因此回滚只影响界面层。

## Open Questions

- 本次先不引入额外毛玻璃/模糊依赖；若未来需要更强的 iOS 玻璃质感，可在不破坏当前主题 token 的前提下追加独立评估。
