## Context

FileFlash-Bridge 当前的移动端主界面集中在单个工作台页面中，服务状态、共享文件、项目内容与帮助入口都聚合在首页。这样的结构适合快速验证核心传输链路，但随着引导、项目管理和后续设置项逐渐增加，首页容易承载过多“非当前任务”信息。

同时，现有 App 文案大多直接散落在 React Native 页面和状态逻辑中，缺少统一的语言资源层，也没有稳定的应用级设置入口。这使得“增加英文界面”不只是替换文本，还意味着要先建立一套可复用的语言选择、状态持久化与页面刷新机制。

本次变更的约束如下：
- 优先复用现有 React Native 共享代码和 `react-native-paper`，避免为了两页 Tab 结构引入不必要的原生依赖。
- 不改变当前局域网传输、服务启动、安全模式、共享导入与项目数据模型。
- 仅支持中文与英文两种语言，首版不要求自动跟随系统语言。
- iOS 与 Android 必须共享同一套导航与语言切换逻辑，仅在安全区、返回键和菜单浮层定位上允许细微差异。

## Goals / Non-Goals

**Goals:**
- 为移动端建立底部 Tab 壳层，提供“首页”和“设置”两个一级入口。
- 保持现有工作台逻辑完整迁移到首页 Tab，不因新增设置页而打断传输主流程。
- 建立 App 级国际化基础设施，让首页、设置页和本次涉及的公共 UI 文案可以在中文 / 英文之间切换。
- 浏览器端门户页面也必须支持国际化，并自动跟随 App 当前语言偏好（无需在网页端单独提供语言切换入口）。
- 在设置页提供清晰的语言设置项，包含语言图标、当前值和点击后可选择“中文 / English”的下拉菜单。
- 在本地持久化用户语言偏好，并在下次启动时恢复。

**Non-Goals:**
- 本次不提供“独立于 App 的网页端语言切换入口”；浏览器端语言必须由 App 端语言偏好自动决定。
- 本次不引入账号体系、云端偏好同步或跨设备共享语言设置。
- 本次不自动读取系统语言并强制覆盖用户手动选择；若后续需要系统语言跟随，可作为独立迭代。
- 本次不重做现有传输链路、权限流程或原生分享能力。

## Decisions

### 1. 使用共享的底部 Tab 壳层，而不是继续在首页堆叠设置入口

移动端将新增一个共享 App Shell，在最外层承载两个底部 Tab：
- 首页：继续承载当前工作台内容与现有业务交互。
- 设置：承载应用级偏好项，首期至少包含语言设置。

这样做的原因：
- 可以把“正在进行的传输工作”与“低频配置项”分开，减轻首页认知负担。
- 设置项不再挤占首页顶部或卡片区，后续新增更多偏好项时也有稳定承载位置。
- iOS 与 Android 共用同一套结构，便于测试和维护。

备选方案：
- 继续在首页顶部或抽屉中塞入设置项。优点是改动小；缺点是首页继续变重，设置能力缺少稳定扩展边界。
- 引入完整的 React Navigation。优点是未来扩展更强；缺点是当前仅两页 Tab，成本高于必要范围，还会引入更多导航依赖与集成工作。

结论：优先基于现有 `react-native-paper` 与共享状态层实现轻量级底部 Tab 壳层。

### 2. 国际化优先采用“共享语言资源 + 顶层语言状态”的纯 JS 方案

本次会建立集中式语言资源层，例如：
- `src/app/i18n/resources.ts`: 定义 `zh-CN` / `en-US` 资源表
- `src/app/i18n/provider.tsx`: 提供当前语言上下文与切换能力
- `src/app/i18n/useI18n.ts`: 为界面层暴露翻译函数

设计目标不是只替换几个按钮，而是为后续持续国际化建立可扩展的边界。首版只要求覆盖本次受影响的 App 内移动端界面。

备选方案：
- 手工在每个组件中写 `locale === 'en' ? ... : ...`。优点是快；缺点是后续维护成本高，难以扩展。
- 引入系统语言探测型原生库。优点是首启可自动匹配；缺点是当前用户明确只需要手动切换中英，且会增加跨平台依赖与原生边界。

结论：先以纯 JS 国际化层为核心，默认语言由本地偏好或应用默认值决定。

### 3. 语言偏好持久化复用现有本地存储边界，不新增原生偏好依赖

语言选择属于轻量级本地偏好，且与首页 / 设置页 UI 强相关。为了减少原生接入成本，本次应优先复用现有本地持久化边界或新增一个共享的偏好文件读写 helper，而不是引入新的原生存储库。

需要持久化的数据非常小：
- `locale`: `'zh-CN' | 'en-US'`

设计要求：
- 读取失败或值非法时，必须回退到默认语言。
- 切换语言后，界面必须立即刷新，不要求重启 App。

备选方案：
- 引入 AsyncStorage 或平台专属偏好存储。优点是常见；缺点是增加额外依赖，而当前需求完全可以由现有边界承载。

结论：复用现有共享存储边界，实现一个最小语言偏好读写层。

### 4. 语言设置项放在设置页中，并采用语言图标 + 菜单浮层

设置页将采用列表式信息架构。语言切换项包含：
- 语言图标
- 标题
- 当前语言值
- 点击后弹出的下拉菜单或等价浮层

当前仅提供两项：
- 中文
- English

原因：
- 与用户要求一致，信息密度低，交互路径清晰。
- `react-native-paper` 已具备 `List.Item`、`Menu`、`Portal` 等适合此场景的共享组件能力。
- 相比原生 Picker，更容易保持 iOS / Android 一致的视觉和交互语义。

备选方案：
- 使用系统 Picker / ActionSheet。优点是平台原生；缺点是双端行为不一致，且视觉风格更难统一。

结论：设置页优先使用 Paper 风格的 item + menu 结构。

### 5. 首页保留当前工作台语义，但挂载到 Home Tab 下

首页 Tab 不应因为新增设置页而改变用户已熟悉的业务心智。迁移后仍需保持：
- 服务控制区可见且可操作
- 共享文件区可继续导入 / 查看
- 当前项目区可继续查看消息和文件

需要特别注意：
- 从设置页返回首页时，不应丢失当前工作台上下文。
- Android 返回键在 Tab 壳层下仍需优先遵循当前页面语义，避免误退出。
- 新增底部 Tab 后，可用垂直空间会变少，首页布局不能因为底栏而造成关键内容不可达。

## Risks / Trade-offs

- [首页垂直空间进一步被底部 Tab 挤占] → 继续保持首页信息分层，避免把设置项留在首页；必要时压缩非核心状态区高度。
- [国际化首轮替换范围过大，容易漏掉散落字符串] → 首版明确以首页 / 设置页 / 本次新增公共 UI 为核心覆盖范围，使用集中资源表逐步收口。
- [语言菜单在小屏设备上遮挡内容或定位不稳定] → 使用 `Portal` / 菜单锚点方案，并分别验证 iOS 与 Android 小屏布局。
- [本地存储异常导致语言状态无法恢复] → 读取失败时回退到默认中文，并保证不会阻断 App 启动。
- [底部 Tab 壳层引入后，首页现有交互回归] → 在任务中单独保留局域网传输、导入共享、项目查看与权限流程回归验证。

## Migration Plan

1. 建立语言资源层与本地语言偏好模型，默认回退到中文。
2. 抽离移动端 App Shell，为首页 / 设置页建立底部 Tab 结构。
3. 将当前工作台挂到首页 Tab 下，并新增设置页。
4. 在设置页加入语言 item、语言图标和中文 / English 菜单。
5. 替换本次涉及页面的硬编码文案，接入语言资源。
6. 在 iOS / Android 上分别验证导航、语言切换、菜单定位和传输回归。

回滚策略：
- 若需要回滚，可先移除底部 Tab 壳层与语言设置入口，保留既有首页结构。
- 即使保留本地语言偏好文件，也不会影响现有传输数据与业务状态。

## Open Questions

- 首版国际化是否只覆盖移动端首页 / 设置页，还是同时补齐引导层与全部提示文案；当前建议先完成移动端主路径，再逐步扩展。
- 设置页后续是否会承载更多应用级选项（例如主题、帮助、关于）；若会，本次列表结构应提前预留扩展空间。
