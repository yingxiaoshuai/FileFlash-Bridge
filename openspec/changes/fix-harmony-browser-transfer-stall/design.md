## Context

FileFlash-Bridge 的核心传输路径是：浏览器访问手机局域网 HTTP 服务，通过门户页上传文件、提交文本，或下载 App 用户显式加入共享列表的文件。当前鸿蒙构建下，门户页点击下载会停在“下载中 0%”，上传也可能无响应；App 侧同步出现 `RangeError: Property storage exceeds 196607 properties`。

现有门户下载逻辑会为完整文件创建分块计划、保存每个下载分片，再用 `Blob(chunks)` 一次性组装完整文件。现有上传分块也会把浏览器 `Blob` 分片转换成 `Uint8Array` 后通过本地服务链路提交。在 Harmony/RNOH/Hermes 环境里，过大的 `Uint8Array` 或跨 JS/Native 边界传递的二进制对象可能被展开为大量数字属性，超过 Hermes property storage 上限后导致运行时异常，继而表现为浏览器按钮无反应、进度停留 0% 或请求 500。

本变更需要保持既有产品边界：浏览器无需安装客户端；浏览器上传文件仍进入 App 内会话存储；只有 App 用户显式共享的文件才允许浏览器下载；二进制传递不得回退到 base64。

## Goals / Non-Goals

**Goals:**

- 在 HarmonyOS 上修复浏览器门户上传/下载卡死，避免触发 Hermes/RNOH property storage 上限。
- 为 JS 层、鸿蒙静态服务 TurboModule、文件访问桥之间的二进制传递定义有界分块策略。
- 让上传、下载和失败状态都能在门户页持续反馈，不再长时间停留在“0%”或无响应。
- 调整 App 与网页端共享下载文案，使“上传到手机”“从手机下载”“保存到本机”在界面上可区分。
- 在 App 共享文件区支持批量选择下载/保存，在浏览器门户共享下载区支持批量选择下载。
- 保持 iOS、Android 共享门户能力一致；非鸿蒙平台继续支持上传、文本提交、共享下载。
- 保持安全边界：不扩大共享目录范围，不自动写入系统公共目录，不在日志中记录文件内容。

**Non-Goals:**

- 不新增云传输、外网中继、账号体系或跨设备发现能力。
- 不把浏览器上传成功改成自动保存到手机公共下载目录或相册。
- 不引入 base64 作为鸿蒙大文件二进制传递方案。
- 不把多个文件打包成 zip 或新增压缩包下载能力；批量下载只编排多个既有文件下载/保存任务。
- 不重做门户视觉系统，只修复传输链路和必要的状态反馈。

## Decisions

1. **鸿蒙二进制桥接采用小块上限，而不是依赖单次大 `Uint8Array`。**

   - 决策：为 Harmony 静态服务与 JS runtime 之间的请求体、响应体、文件桥读写设置保守分块上限，使每次跨边界字节载荷低于 Hermes/RNOH 属性展开风险阈值。JS 层处理上传 part、下载 range、文件桥 append/read 时都按该上限循环。
   - 理由：问题根源是大二进制对象在运行时边界被展开后超过属性数量限制。降低单次跨边界对象规模，可以直接避免崩溃。
   - 备选：继续提高浏览器 chunkSize。该方案会加重崩溃风险，不采用。
   - 备选：改用 base64。用户已明确禁止，且会增加体积和内存压力，不采用。

2. **门户上传使用浏览器原生 Blob/stream 能力优先，鸿蒙链路只接收有界 part。**

   - 决策：浏览器侧继续按文件分片上传，但每个 HTTP part 的大小必须受控；如 runtime 对 `Uint8Array` 传递不稳定，优先让 fetch/XMLHttpRequest 直接发送 `Blob` 分片或 `ArrayBuffer`，由本地服务在鸿蒙侧拆成更小 Native/JS 桥接片段写入。
   - 理由：浏览器原生二进制对象不需要先在门户 JS 中展开为数字属性；鸿蒙服务内部再用更小的桥接片段落盘。
   - 备选：一次性上传整个文件。会导致内存峰值和桥接对象过大，不采用。

3. **门户下载改为增量写入结果容器，避免保存完整分片数组。**

   - 决策：下载共享文件时按 range/part 顺序或有限并发拉取，进度按已确认字节推进；门户不得维护超大 `downloadedChunks` 数组或完整 `chunkPlan` 对象集合。浏览器支持 File System Access API 时可直接写入文件；不支持时使用受控缓冲并限制适用文件大小，超限时给出可理解的提示或降级策略。
   - 理由：当前“先保存所有分片再 Blob 合并”的模型会把大文件完整压在浏览器内存里，也会产生大量 JS 对象。增量写入能降低内存峰值和对象数量。
   - 备选：下载完成后再一次性合成 Blob。小文件可保留，大文件不采用。

4. **错误必须跨 App 与门户同步暴露。**

   - 决策：鸿蒙文件桥、静态服务和门户页统一返回错误分类，例如桥接载荷过大、文件读取失败、存储空间不足、网络中断、服务停止。门户必须中止当前任务、清理状态、允许重试。
   - 理由：当前运行时崩溃会让浏览器看到“无响应”，不利于用户定位和重试。
   - 备选：只在 App 日志记录。无法解决浏览器用户体验，不采用。

5. **批量下载复用单文件下载/保存链路。**

   - 决策：App 端共享文件区增加选择模式，用户选择多个共享文件后，系统按顺序或受控并发调用现有显式导出/保存流程；网页端共享下载区增加选择控件和批量下载按钮，批量操作逐个复用现有单文件下载逻辑。
   - 理由：复用单文件路径可以继承现有权限、安全边界、失败提示和大文件分块策略，避免为批量操作引入新的文件格式或存储语义。
   - 备选：将多个文件打包为 zip 后一次下载。该方案会引入额外压缩、临时存储和大文件内存压力，不属于本次修复范围。
   - 备选：App 端批量操作直接把文件自动写入公共下载目录。该方案违反显式保存/导出边界，不采用。

6. **文案以方向和主体为中心。**

   - 决策：App 与门户文案必须明确“上传到手机”“从手机下载”“手机共享文件”“下载选中”等动作方向。App 端按钮可以根据平台实际能力落到“保存/导出”，但用户任务入口命名应围绕下载选中文件。
   - 理由：当前“共享文件/导出/下载”混用容易让用户误解文件所在位置，尤其是浏览器与手机双端同时操作时。

7. **平台差异只收敛在 runtime 适配层。**

   - 决策：React Native 共享层负责统一分块策略、任务状态和门户 HTML；鸿蒙层负责 Native socket/文件桥的安全字节边界；iOS/Android 继续沿用现有 HTTP/file 实现，但必须通过同一组传输测试。
   - 理由：核心产品体验应跨平台一致，鸿蒙的限制应封装在平台适配层。

## Risks / Trade-offs

- [Risk] 更小的鸿蒙桥接分块会增加请求处理次数，降低峰值吞吐。 → Mitigation：只在跨 Harmony JS/Native 边界使用保守上限，浏览器 HTTP 层可保留更大的逻辑分片；通过进度反馈减少用户误判。
- [Risk] 部分移动浏览器不支持 File System Access API，无法真正流式保存下载文件。 → Mitigation：小文件继续 Blob 保存；大文件给出明确降级提示，或使用按平台可用能力的增量写入方案。
- [Risk] 有限并发下载如果仍然缓存多个 part，内存峰值可能过高。 → Mitigation：限制并发和每 part 大小，并在 Harmony 浏览器链路优先使用顺序或低并发下载。
- [Risk] 批量下载会同时触发多个系统保存/浏览器下载提示，用户可能感到打断。 → Mitigation：默认顺序执行或限制并发，并在 UI 上显示已选择数量、当前进度和失败项。
- [Risk] 修改传输协议可能影响现有 Android/iOS 浏览器兼容性。 → Mitigation：保持原有 `/api/upload`、`/api/upload/begin`、`/api/upload/part`、`/api/upload/finish`、`/api/shared` 与下载接口语义不变，仅收紧实现边界；增加回归验证。
- [Risk] 服务端进程崩溃后浏览器请求仍可能悬挂。 → Mitigation：门户为上传/下载设置超时、abort 和重试状态清理；App 侧捕获桥接异常并返回结构化 5xx 错误。

## Migration Plan

1. 引入共享的传输分块常量，区分浏览器逻辑 chunk 与 Harmony JS/Native 桥接 chunk。
2. 修改 Harmony 静态服务请求/响应处理，使进入 RN JS 的 `bodyBytes` 与返回给 socket 的 `bodyBytes` 都经过有界拆分或可流式发送路径。
3. 修改上传保存路径，确保大文件按小块追加写入 App 内部存储，JSON 文件也按字节处理而不是文本特殊路径。
4. 修改门户下载实现，避免大文件完整分片数组与一次性 Blob 合并，并补齐超时、失败提示和状态清理。
5. 修改 App 与门户共享下载文案，增加批量选择、全选/清空、下载选中和空选择提示。
6. 分别验证 Harmony、Android、iOS：小 JSON 上传、大文件上传、共享大文件下载、批量下载、服务停止/网络中断失败反馈。

Rollback 策略：保留现有接口路由和响应格式，必要时可以回退到旧的门户 HTML 与 HTTP runtime 实现；但鸿蒙旧实现已知会触发大文件卡死，只能作为临时回退。

## Open Questions

- 鸿蒙目标浏览器是否支持 File System Access API，或是否需要完全依赖 Blob 下载降级。
- Harmony/RNOH 当前版本对 `Uint8Array`、`ArrayBuffer`、Plain object 的序列化上限是否有官方可配置项；实现上先按不可配置处理。
- App 端批量下载在不同平台上是否允许一次选择目录后保存多个文件；若系统限制逐个保存，则以顺序弹出系统保存流程作为 V1 行为。
