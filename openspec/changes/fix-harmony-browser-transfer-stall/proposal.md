## Why

鸿蒙构建下，浏览器门户点击共享文件下载后停在“下载中 0%”，上传文件也可能无响应或失败；同时 App 侧出现 `RangeError: Property storage exceeds 196607 properties`，说明当前大块二进制在 Harmony/RNOH/Hermes 链路中被展开成过多 JS 属性，导致运行时崩溃或请求卡死。

这个问题直接阻断 FileFlash-Bridge 的核心场景：其它设备通过浏览器向鸿蒙手机上传文件，或从鸿蒙手机下载共享文件。需要把鸿蒙浏览器传输链路改成可控分块、可恢复失败、不会一次性缓存整文件的实现，并同步补齐 App 与网页端对“下载手机共享文件”的文案和批量选择下载能力。

## What Changes

- 调整鸿蒙本地传输服务的上传/下载二进制桥接策略，确保进入 React Native JS 运行时的单次字节载荷受控，避免触发 Hermes/RNOH property storage 上限。
- 调整浏览器门户的大文件上传流程，避免把大文件或过大的分片作为会被枚举展开的 `Uint8Array` 对象跨运行时传递；失败时必须展示明确原因，而不是按钮无响应。
- 调整浏览器门户的大文件下载流程，避免预先创建超大 chunk 数组、缓存所有分片后再一次性合成 Blob；下载进度必须推进，失败必须中止并提示。
- 调整 App 共享文件区文案，明确区分“添加到手机共享列表”和“下载/保存选中文件”，避免用户把共享、导出和浏览器下载混为一谈。
- 为 App 共享文件区增加批量选择下载/保存设计：用户可以进入选择模式、选择多个共享文件、全选/清空，并一次性触发选中文件的显式下载/保存流程。
- 为浏览器门户共享下载区增加批量选择下载设计：浏览器用户可以选择多个手机共享文件、全选/清空，并批量发起下载；单个文件仍保留独立下载入口。
- 保持“不使用 base64 传递二进制”的约束；二进制仍以 `Uint8Array`、`ArrayBuffer`、Blob/stream 或平台等价字节结构处理。
- 不引入新的跨端产品能力；本次变更修复现有局域网浏览器传输在 HarmonyOS 上的稳定性。

## Capabilities

### New Capabilities

无。

### Modified Capabilities

- `local-transfer-server`: 明确鸿蒙本地传输服务在上传保存、共享下载和 JS/Native 字节桥接时，必须使用有界分块，不能因单次二进制对象过大导致 Hermes/RNOH 属性存储溢出、服务崩溃或浏览器请求悬挂；同时 App 工作台共享文件区必须支持批量选择下载/保存。
- `web-transfer-portal`: 明确浏览器门户上传/下载大文件时不得一次性缓存整文件或维护超大分片对象集合；必须持续更新进度，并在鸿蒙链路失败时显示可理解的失败提示；同时门户共享下载区必须支持批量选择下载。

## Impact

- 影响共享代码：`src/modules/portal/portalDocument.ts`、`src/modules/localization/i18n.tsx`、App 工作台共享文件区、传输服务控制器、HTTP runtime 适配层和文件分块策略。
- 影响鸿蒙平台代码：Harmony 本地静态服务 TurboModule、请求体/响应体二进制传递方式、文件访问桥与大文件读写分块边界。
- iOS/Android 影响：共享门户逻辑会收敛到更稳健的分块上传/下载模型；非鸿蒙平台不得出现功能退化，仍需支持浏览器上传文件、提交文本、下载共享文件。
- 安全与隐私：仍只暴露用户显式共享的文件列表；浏览器上传内容仍保存到 App 内部会话存储，不自动写入系统公共目录；失败日志不得泄露文件内容。
- 依赖影响：优先复用现有 React Native 与鸿蒙 Native Module，不默认引入新的第三方原生库。
