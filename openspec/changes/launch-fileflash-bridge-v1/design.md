## Context

FileFlash-Bridge V1.0 需要在一套 React Native 代码基础上，同时交付 iOS 与 Android 的“浏览器与手机之间的近场文件与文本交换”能力：既包括**浏览器向手机投递**，也包括用户在 **App 内显式选定**后、由**浏览器拉取（下载）**手机上的共享文件。当前仓库还没有既有能力约束，因此本次设计既是首版产品架构定义，也是后续功能扩展的基线。

该方案同时涉及 React Native 共享业务层、本地网络服务、浏览器访问页面、文件系统读写（入站接收与出站共享）、文本接收、二维码展示和基础安全控制，属于跨模块且带有平台差异和安全约束的设计型变更。V1.0 的重点是在打通“手机开启服务 -> 其他设备浏览器接入 -> 上传 / 下载 / 文本提交”的完整闭环的同时，确保服务在后台/息屏场景仍可持续可用。

约束条件如下：
- 业务逻辑优先使用 React Native 共享代码，避免 iOS/Android 双端能力漂移。
- 文件访问、局域网发现、网络权限和本地服务器生命周期必须分别考虑 iOS 与 Android 差异。
- 传输链路默认运行在局域网，仍需提供 `key` 访问控制、连接数限制与手动停止能力，降低泄露风险。
- V1.0 必须支持后台/息屏保活，避免用户切后台或锁屏后服务不可用。
- 用户侧不具备创建或部署 HTTPS 的能力，且项目方无法提供可用证书；V1.0 必须仅基于 HTTP 交付可用体验。
- 浏览器仅能下载用户在 App 内**主动加入共享列表**的文件，不得默认暴露整机目录或全盘文件列表。

## Goals / Non-Goals

**Goals:**
- 提供一个由 React Native 驱动的本地传输服务管理层，统一负责服务启停、地址生成、二维码展示、连接状态和错误提示。
- 提供浏览器访问的响应式门户页，支持文件/文件夹上传、文本粘贴提交、**用户选定文件的列表展示与下载**，以及上传/下载的实时进度反馈。
- 为 iOS 与 Android 定义统一的**会话内入站存储**抽象：浏览器传入的文件与文本项目数据**持久化保存在本机 App 内部**（按会话/项目隔离），**不自动**写入相册、系统「下载」或用户未主动操作时的外部可见目录；用户可在会话内对需要的文件执行**显式「保存到…」**导出。
- 提供手机端文本接收区：按「项目 / 会话」归档（类似 AI 聊天），同一项目内可包含多次浏览器文本提交；用户可**新建项目**以开始下一轮分享，并可在项目内对单条消息复制/删除或删除整项；**删除会话/项目前**须提示将清除该会话下的文件，并引导用户在删除前进会话保存所需内容。
- 提供首版安全控制，包括“简单模式 / 安全模式”切换、连接设备数量限制与服务手动停止能力。
- 服务在 App 切到后台或设备息屏后，仍应保持可访问，并具备被系统干预时的恢复与提示能力。
- 定义真机验证路径，覆盖公共 Wi-Fi 与热点两种网络模式、上传、**大文件分块下载**与文本提交成功率及异常恢复。

**Non-Goals:**
- 不在 V1.0 中实现剪贴板同步、传输历史、在线预览等 V2.0 功能。
- 不支持 HTTPS，不提供证书生成、安装、分发或浏览器信任链引导。
- 不提供跨互联网远程传输能力，范围仅限局域网或手机热点直连网络。
- 不要求浏览器像文件管理器一样浏览手机任意目录；**不提供** WebDAV、全盘索引或未经用户逐次授权的全量文件列表。

## Decisions

### Decision 1: 采用“React Native 共享控制层 + 轻量原生服务适配层”的分层架构

选择：
- 共享层负责服务配置、状态机、UI、二维码、Web 资产挂载、权限编排和错误展示。
- 原生适配层只处理 React Native 无法稳定覆盖的能力，例如本地 HTTP 服务绑定、平台文件句柄访问、系统权限桥接和网络接口枚举。

原因：
- 这能最大化复用双端代码，同时把 iOS/Android 差异限制在少量适配接口中。
- 纯原生双实现会造成能力漂移和后期维护成本上升。

替代方案：
- 纯 React Native 社区库拼装。问题是本地服务、文件句柄和部分权限行为在双端一致性上风险较大。
- 双端完全原生实现。问题是交付周期长，不符合首版快速成型目标。

### Decision 2: V1.0 以本地 HTTP 服务承载入站投递与「用户选定」出站下载，不引入 WebDAV

选择：
- 浏览器访问、文件上传、**用户已共享文件的列表与分块下载**、文本提交和状态查询全部基于 HTTP 路由实现（同源）。
- 手机侧**仅暴露用户在 App 内明确选择纳入「当前共享」的文件**（可多条）；不提供任意目录浏览、不提供 WebDAV、不提供文件夹打包下载作为目录暴露的替代（若需多文件，由用户在 App 内逐条或批量加入共享列表，门户仅展示该列表）。

原因：
- 浏览器天然兼容 HTTP；在可控共享集合内提供下载，可补齐「电脑取回手机文件」路径，同时把暴露面限制在用户意图范围内。
- WebDAV 或全盘列表会显著扩大攻击面与测试成本；显式共享列表 + 同源 key 策略更易做安全边界。

替代方案：
- 完全不做出站能力。优点是实现更简单，但无法满足「由 App 选文件、网页下载」的产品诉求。
- 首版 WebDAV 或目录浏览。优点是通用，但安全与双端权限成本过高。

### Decision 3: 大文件下载采用分块传输，并对单分块失败重试设硬上限

选择：
- 对超过阈值的大文件，浏览器门户必须通过**分块**方式拉取（例如固定大小分块接口或 HTTP Range 等等价机制），避免单次响应过大导致超时或内存压力；具体分块大小可配置或约定默认值。
- **失败重试**：对每个分块的请求在失败时自动重试；**同一分块在首次请求失败后，最多再重试 3 次**（即单分块至多 **4** 次传输尝试）。若某分块用尽重试仍失败，**必须**中止当前文件的整体下载流程，并向用户明确提示**传输失败**（含失败分块或错误原因的可读说明）。
- 重试责任落在**浏览器门户侧**（客户端逐块请求与重试）；服务端返回明确错误码/状态，便于区分可重试与不可重试（如 404/403 可视为整文件失败）。

原因：
- 局域网仍可能出现瞬时断连或缓冲区问题，分块 + 有限次重试能提升大文件成功率，同时避免无限重试挂死 UI。
- 硬上限保证可预期的失败语义，便于测试与用户理解。

替代方案：
- 单次整文件 GET。实现简单但在大文件与弱网下失败率高。
- 无上限重试。可能导致页面长时间无结果，难以区分卡死与传输中。

### Decision 4: 浏览器门户页面由 App 内嵌静态资源提供，接口与静态页同源部署

选择：
- 将响应式 Web 页面作为 App 资源内嵌，并由本地服务直接提供 HTML/CSS/JS 与 API。
- 上传、下载与文本提交状态通过同源接口轮询、分块进度聚合或流式状态返回展示。

原因：
- 这样可以避免额外远端托管，保证离线和局域网内即可工作。
- 同源部署能简化浏览器跨域问题与首版安全策略。

替代方案：
- 使用远端页面托管。问题是不符合“离线局域网可用”的产品价值。

### Decision 5: 入站文件与会话数据落在 App 内部并持久化；压缩存小、跳过大文件；导出须用户显式操作

选择：
- 浏览器上传的文件及文本会话的元数据与内容**全部持久化保存在手机本机**，默认位于 **App 托管存储**（按**会话 / 项目**隔离），**不自动**复制到 App 外部的系统目录（如公共下载、相册等）。用户若需在外部长期使用或备份，必须在 App 内**进入对应会话**，通过**显式「保存到…」/ 分享导出**等流程写入用户选定位置（此时再按需走系统文件选择器 / SAF 等能力）。
- **压缩策略**：对接收的入站文件在落盘时进行**无损压缩封装**（如按文件打包为 zip 成员或等价形式），以降低被系统或其它应用**误当作可执行内容直接触发**的风险，并作为针对恶意投递的纵深措施之一（**不替代**访问控制与用户警觉）。对**超过实现约定阈值的大文件**可**跳过压缩**，以避免耗时、内存与电量问题；阈值可配置。
- **删除会话 / 项目**：用户确认删除前必须展示**不可忽略的提示**，说明**删除将清除该会话下的会话数据及关联文件**；并提示**如有需要保留的文件，请先进入该会话将其保存到外部**。确认后才执行删除与存储回收。

原因：
- 局域网 HTTP 投递存在被滥用投递可执行文件或恶意内容的风险；默认留在 App 内 + 压缩封装可提高「无意双击即执行」的门槛，同时避免用户误以为文件已自动出现在系统文件管理器中。
- 大文件跳过压缩是工程上的必要折中。
- 持久化与会话绑定删除语义一致，需通过文案避免用户误删后数据不可恢复。

替代方案：
- 入站直接写入用户自选目录。优点是直观，但自动落盘到外部会放大恶意文件暴露面，且与「删除会话清文件」模型冲突。
- 全盘不压缩。实现简单，但安全纵深较弱。

### Decision 6: 浏览器提交的文本内容先进入手机端接收区，不直接强制覆盖系统剪贴板

选择：
- 浏览器门户提供文本输入 / 粘贴区，通过 HTTP 接口把文本提交到 App。
- App 侧以**项目（会话）**为顶层单位组织文本接收（类似 AI 聊天）：每个项目下按时间线保存**多次**浏览器成功提交的文本、来源时间与状态；**并非**「每次提交即一条顶层记录」。须维护**当前活跃项目**，浏览器新提交默认追加到该项目内的消息列表。用户可**新建项目**再开始下一轮分享，历史项目与其下全部消息保留直至用户删除；支持对项目内**单条消息**复制与删除，以及删除整个项目。
- 文本提交成功后可以提示用户“一键复制到手机剪贴板”，但不在后台静默覆盖系统剪贴板。

原因：
- 用户需求是“浏览器可以把粘贴内容发到手机”，而不是强依赖底层剪贴板同步协议。
- 直接覆盖系统剪贴板会带来意外覆盖、后台行为不稳定和平台差异问题。

替代方案：
- 浏览器提交后立即写入系统剪贴板。优点是更接近实时粘贴，但会增加平台行为不确定性和误覆盖风险。

### Decision 7: 安全能力采用“简单模式 / 安全模式（URL key）”的 HTTP 策略

选择：
- 提供两种模式：
  - 简单模式：用户只需在浏览器输入 URL（或扫码）即可访问传输门户页。
  - 安全模式：URL 中必须包含一个 `key` 参数（或路径片段）作为访问凭证；缺少或错误的 key 视为未授权。
- `key` 由 App 生成并可一键刷新，刷新后旧 key 立即失效。
- 两种模式都只使用 HTTP，不涉及证书与 HTTPS。

原因：
- 用户侧无法创建 HTTPS，且项目方无法提供证书；将 HTTPS 作为能力会导致不可落地或体验不可控。
- 局域网传输的首要风险是未授权访问；将访问凭证收敛为 URL 中的 `key` 能在不引入证书的情况下提升安全性，并减少额外输入步骤。

替代方案：
- 继续使用“访问密码”弹窗。问题是跨设备输入成本高、容易被忽略，且分享/扫码链路上不如 key 直达。
- 强制 HTTPS。问题是无法满足证书约束，无法交付。
- 完全不做安全控制。问题是风险过高。

### Decision 8: 服务生命周期以“显式启停 + 后台保活”为核心能力

选择：
- 用户进入传输页并完成必要权限后，显式启动服务（入站默认不再依赖「预先授权外部接收目录」作为启动前置条件，导出仍走会话内显式操作）。
- App 在前台与后台均保持服务运行，并展示状态、地址、二维码和活动连接数。
- Android 侧以“前台服务 + 常驻通知”的方式保活传输服务，避免后台限制导致服务被挂起。
- iOS 侧以原生后台执行能力（例如后台任务窗口）尽可能维持本地服务可用，并在被系统干预时提供自动恢复/一键恢复能力与清晰提示。

原因：
- 用户核心场景要求“切后台/息屏仍可传”，保活是 V1.0 的必要能力而非增强项。
- Android 与 iOS 的后台策略差异明显，需要在统一体验目标下分别采用平台推荐做法落地。

替代方案：
- 仅前台可用。问题是无法满足产品核心诉求。

## Risks / Trade-offs

- [iOS 局域网权限或本地网络发现受限] → 首次启动服务前集中弹出权限说明，并在失败时提供重试与系统设置引导。
- [Android SAF / 系统文件选择器在「导出」路径上的 URI 行为复杂] → 将会话内导出收敛到统一网关并在真机上覆盖多品牌机型验证。
- [大文件或整目录上传中断] → 采用流式写入、失败重试和清晰的上传状态提示，必要时分批保存临时产物。
- [大文件下载分块失败或反复超时] → 分块拉取 + 单分块「首次 + 至多 3 次重试」策略；用尽后整文件失败并提示用户检查网络或重新共享后重试。
- [App 内部存储空间不足或写入失败] → 在写入前检查可用空间与权限，并对浏览器端和手机端同时返回失败原因；提示用户清理会话或导出后删除。
- [文本提交内容过大或包含敏感数据] → 限制单次文本长度；对「项目数量、单项目内消息条数或总占用」设置合理上限（可配置），并提供按消息/按项目删除能力，避免敏感内容无限堆积。
- [用户误以为接收文件已出现在系统「下载」或相册] → 接收成功提示与首次引导中明确「默认仅在 App 内会话中」；导出入口放在会话内。
- [压缩被误解为杀毒或绝对安全] → 文案标明压缩仅为降低误执行风险的存储策略之一，**不替代** key、连接限制与用户判断。
- [HTTP 明文传输在不受信任局域网的泄露风险] → 默认提示“仅建议在可信 Wi-Fi/热点使用”；提供安全模式（URL key）降低未授权访问风险，但不承诺链路加密。
- [热点模式下不同机型 IP 和网卡行为不一致] → 增加网络接口探测与手动刷新地址能力，并在真机验证中覆盖热点场景。
 - [URL key 被截屏/转发导致泄露] → 提供一键刷新 key；可选增加“自动过期”作为后续增强（不作为 V1.0 强制项）。
- [iOS 后台执行窗口与系统调度不确定] → 采用“尽可能持续 + 自动/一键恢复 + 明确状态提示”的组合策略，并将真实机型与系统版本覆盖纳入验收。
- [Android 前台服务带来常驻通知与耗电] → 提供清晰的“停止服务”入口与状态提示，避免用户误以为服务已关闭。

## Migration Plan

1. 初始化 React Native 项目骨架与共享目录结构，预留 `service`, `file-access`, `portal`, `security` 模块边界。
2. 接入本地 HTTP 服务与 Web 静态资源分发能力，先打通地址访问与基础健康检查。
3. 接入**会话内入站存储**（持久化、按项目隔离、小文件压缩/大文件跳过）、**App 内共享文件选择与会话内共享列表**、文本接收区与会话删除确认文案，在双端完成文件读写、导出能力与文本落地。
4. 实现浏览器门户的上传、**共享文件列表与分块下载（含分块失败重试与整文件失败提示）**、文本提交与进度展示，再补充二维码、简单/安全模式与连接控制。
5. 在 iOS 与 Android 真机上分别验证公共 Wi-Fi、热点、权限拒绝、大文件上传、**大文件下载分块与重试用尽失败路径**、文本提交和服务中断恢复。
6. 增强后台保活与状态恢复，确保浏览器向手机的主路径在切后台/息屏时仍尽量可用。

回滚策略：
- 若会话内落盘或压缩链路不稳定，可先关闭压缩（仍以 App 内存储接收）或暂时关闭目录上传，仅保留单文件上传与文本提交作为缩小范围的临时回退。
- 若安全模式或连接限制引发兼容问题，可关闭对应开关，不影响简单模式主路径。

## Open Questions

- V1.0 是否要求浏览器端支持整文件夹上传，还是允许在不支持目录上传的浏览器中回退为多文件上传。
- 入站压缩阈值默认值与是否对特定类型（如已是压缩包）跳过二次压缩。
- 浏览器提交的文本内容是否需要“自动写入系统剪贴板”，还是保持“进入接收区后由用户手动复制”的模式。
- 安全模式的 `key` 是否需要默认启用，以及是否需要提供可配置的自动过期时间（例如 10 分钟 / 1 小时）。
