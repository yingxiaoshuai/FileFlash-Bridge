## Context

FileFlash-Bridge V1.0 需要在一套 React Native 代码基础上，同时交付 iOS 与 Android 的局域网文件互传能力。当前仓库还没有既有能力约束，因此本次设计既是首版产品架构定义，也是后续功能扩展的基线。

该方案同时涉及 React Native 共享业务层、本地网络服务、浏览器访问页面、文件系统权限、二维码展示和基础安全控制，属于跨模块且带有平台差异和安全约束的设计型变更。V1.0 的重点是打通“手机开启服务 -> 其他设备浏览器接入 -> 双向传输文件”的完整闭环，而不是一次性覆盖所有后台保活和高级协作能力。

约束条件如下：
- 业务逻辑优先使用 React Native 共享代码，避免 iOS/Android 双端能力漂移。
- 文件访问、局域网发现、网络权限和本地服务器生命周期必须分别考虑 iOS 与 Android 差异。
- 传输链路默认运行在局域网，仍需提供密码、连接数限制与手动停止能力，降低泄露风险。
- V1.0 不把锁屏后持续保活作为承诺能力，避免在首版中引入高风险的后台运行策略。
- 用户侧不具备创建或部署 HTTPS 的能力，且项目方无法提供可用证书；V1.0 必须仅基于 HTTP 交付可用体验。
- 必须支持从系统文件管理器“长按文件 -> 分享/发送”直接选择本 App 作为入口，完成“把该文件发给浏览器端”的主路径。

## Goals / Non-Goals

**Goals:**
- 提供一个由 React Native 驱动的本地传输服务管理层，统一负责服务启停、地址生成、二维码展示、连接状态和错误提示。
- 提供浏览器访问的响应式传输页，支持文件浏览、单文件下载、文件夹打包下载、文件/文件夹上传和实时进度反馈。
- 为 iOS 与 Android 定义统一的文件访问抽象，屏蔽目录选择、权限申请、保存路径与 URI 差异。
- 提供首版安全控制，包括“简单模式 / 安全模式”切换、连接设备数量限制与服务手动停止能力。
- 定义真机验证路径，覆盖公共 Wi-Fi 与热点两种网络模式、上传下载成功率和异常恢复。

**Non-Goals:**
- 不在 V1.0 中承诺锁屏后长期后台保活。
- 不在 V1.0 中实现剪贴板同步、传输历史、在线预览等 V2.0 功能。
- 不支持 HTTPS，不提供证书生成、安装、分发或浏览器信任链引导。
- 不提供跨互联网远程传输能力，范围仅限局域网或手机热点直连网络。

## Decisions

### Decision 1: 采用“React Native 共享控制层 + 轻量原生服务适配层”的分层架构

选择：
- 共享层负责服务配置、状态机、UI、二维码、Web 资产挂载、权限编排和错误展示。
- 原生适配层只处理 React Native 无法稳定覆盖的能力，例如本地 HTTP 服务绑定、平台文件句柄访问、系统权限桥接和网络接口枚举。

原因：
- 这能最大化复用双端代码，同时把 iOS/Android 差异限制在少量适配接口中。
- 纯原生双实现会造成能力漂移和后期维护成本上升。

替代方案：
- 纯 React Native 社区库拼装。问题是本地服务、文件句柄和部分权限行为在双端一致性上风险较大。
- 双端完全原生实现。问题是交付周期长，不符合首版快速成型目标。

### Decision 2: V1.0 以本地 HTTP 服务为主协议，服务层预留 WebDAV 扩展边界

选择：
- 浏览器访问、文件列表、上传下载和文件夹打包全部基于 HTTP 路由实现。
- 服务接口层定义可扩展协议适配器，为后续补充 WebDAV 能力预留边界，但 V1.0 的可交付主路径以 HTTP 为准。

原因：
- 浏览器天然兼容 HTTP，能最快满足“无需安装 App”的核心价值。
- WebDAV 在不同客户端和移动端运行时上的兼容性与测试成本更高，直接作为首版强承诺会增加落地风险。

替代方案：
- 首版同时完整支持 HTTP 和 WebDAV。优点是协议更通用，但需要额外解决认证、目录映射和多客户端兼容问题。

### Decision 3: 浏览器门户页面由 App 内嵌静态资源提供，接口与静态页同源部署

选择：
- 将响应式 Web 页面作为 App 资源内嵌，并由本地服务直接提供 HTML/CSS/JS 与 API。
- 上传和下载进度通过同源接口轮询或流式状态返回展示。

原因：
- 这样可以避免额外远端托管，保证离线和局域网内即可工作。
- 同源部署能简化浏览器跨域问题与首版安全策略。

替代方案：
- 使用远端页面托管。问题是不符合“离线局域网可用”的产品价值。

### Decision 4: 文件访问使用统一 File Access Gateway，按平台映射真实目录能力

选择：
- JS 层定义统一接口：目录授权、列出文件、读取文件流、写入上传内容、打包文件夹、删除临时文件。
- iOS 优先基于应用可访问容器与文档选择器映射授权目录。
- Android 优先基于 Storage Access Framework 与内容 URI 访问用户授权目录。

原因：
- 双端文件权限模型差异大，若直接在业务层耦合平台细节，后续所有功能都难以扩展。
- 统一抽象后，上传下载流程、错误提示和测试策略都可以共享。

替代方案：
- 直接固定应用沙盒目录。优点是实现简单，但无法满足用户从常用文件目录中自由传输的需求。

### Decision 5: 安全能力采用“密码保护优先，HTTPS 按可用性启用”的策略

选择：
- 提供两种模式：
  - 简单模式：用户只需在浏览器输入 URL（或扫码）即可访问传输门户页。
  - 安全模式：URL 中必须包含一个 `key` 参数（或路径片段）作为访问凭证；缺少或错误的 key 视为未授权。
- `key` 由 App 生成并可一键刷新，刷新后旧 key 立即失效。
- 两种模式都只使用 HTTP，不涉及证书与 HTTPS。

原因：
- 用户侧无法创建 HTTPS，且项目方无法提供证书；将 HTTPS 作为能力会导致不可落地或体验不可控。
- 局域网传输的首要风险是未授权访问；将访问凭证收敛为 URL 中的 `key` 能在不引入证书的情况下提升安全性，并减少额外输入步骤。

替代方案：
- 继续使用“访问密码”弹窗。问题是跨设备输入成本高、容易被忽略，且分享/扫码链路上不如 key 直达。
- 强制 HTTPS。问题是无法满足证书约束，无法交付。
- 完全不做安全控制。问题是风险过高。

### Decision 6: 服务生命周期以显式启停和可见状态为核心，不默认承诺后台持续可用

选择：
- 用户进入传输页并授权目录后，显式启动服务。
- App 在前台时保持服务运行并展示状态、地址、二维码和活动连接数。
- 当系统限制导致后台服务中断时，App 必须在回到前台后告知用户并允许一键恢复。

原因：
- iOS 与 Android 的后台执行限制差异明显，首版若承诺长时保活，稳定性风险过高。
- 可见状态机能帮助用户理解“为什么此刻无法访问”。

替代方案：
- 首版即强推后台保活。问题是开发复杂度、耗电和审核风险都会明显增加。

## Risks / Trade-offs

- [iOS 局域网权限或本地网络发现受限] → 首次启动服务前集中弹出权限说明，并在失败时提供重试与系统设置引导。
- [Android SAF 目录授权后 URI 行为复杂] → 把目录访问全部收敛到统一网关并在真机上覆盖多品牌机型验证。
- [文件夹打包下载占用较高内存或磁盘] → 采用临时压缩文件和流式下载，完成后及时清理缓存。
- [大文件上传/下载中断] → 下载侧支持 Range 断点续传；上传侧至少提供失败重试与用户可感知的中断提示。
- [HTTP 明文传输在不受信任局域网的泄露风险] → 默认提示“仅建议在可信 Wi-Fi/热点使用”；提供安全模式（URL key）降低未授权访问风险，但不承诺链路加密。
- [热点模式下不同机型 IP 和网卡行为不一致] → 增加网络接口探测与手动刷新地址能力，并在真机验证中覆盖热点场景。
 - [URL key 被截屏/转发导致泄露] → 提供一键刷新 key；可选增加“自动过期”作为后续增强（不作为 V1.0 强制项）。

## Migration Plan

1. 初始化 React Native 项目骨架与共享目录结构，预留 `service`, `file-access`, `portal`, `security` 模块边界。
2. 接入本地 HTTP 服务与 Web 静态资源分发能力，先打通地址访问与基础健康检查。
3. 接入文件访问网关，在双端完成目录授权、文件枚举和文件读写。
4. 实现浏览器门户的下载、上传与进度展示，再补充二维码、简单/安全模式与连接控制。
5. 在 iOS 与 Android 真机上分别验证公共 Wi-Fi、热点、权限拒绝、大文件传输和服务中断恢复。
6. 增强系统分享入口（从系统文件管理器分享给 App）与“快速发送”体验，确保主路径可用。

回滚策略：
- 若服务层或文件访问层不稳定，可先关闭上传能力，仅保留下载能力作为缩小范围的临时回退。
- 若安全模式或连接限制引发兼容问题，可关闭对应开关，不影响简单模式主路径。

## Open Questions

- V1.0 是否要求浏览器端支持整文件夹上传，还是允许在不支持目录上传的浏览器中回退为多文件上传。
- iOS 侧首版允许暴露哪些目录给浏览器访问，是仅用户显式授权目录，还是允许映射应用内导入区。
- WebDAV 扩展在 V1.0 是否需要保留实验性入口，还是仅保留内部接口不对外暴露。
- 安全模式的 `key` 是否需要默认启用，以及是否需要提供可配置的自动过期时间（例如 10 分钟 / 1 小时）。
