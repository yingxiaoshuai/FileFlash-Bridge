## Context

FileFlash-Bridge V1.0 需要在一套 React Native 代码基础上，同时交付 iOS 与 Android 的“浏览器向手机投递文件和文本内容”能力。当前仓库还没有既有能力约束，因此本次设计既是首版产品架构定义，也是后续功能扩展的基线。

该方案同时涉及 React Native 共享业务层、本地网络服务、浏览器访问页面、文件系统写入、文本接收、二维码展示和基础安全控制，属于跨模块且带有平台差异和安全约束的设计型变更。V1.0 的重点是在打通“手机开启服务 -> 其他设备浏览器接入 -> 向手机上传文件 / 提交文本内容”的完整闭环的同时，确保服务在后台/息屏场景仍可持续可用。

约束条件如下：
- 业务逻辑优先使用 React Native 共享代码，避免 iOS/Android 双端能力漂移。
- 文件访问、局域网发现、网络权限和本地服务器生命周期必须分别考虑 iOS 与 Android 差异。
- 传输链路默认运行在局域网，仍需提供 `key` 访问控制、连接数限制与手动停止能力，降低泄露风险。
- V1.0 必须支持后台/息屏保活，避免用户切后台或锁屏后服务不可用。
- 用户侧不具备创建或部署 HTTPS 的能力，且项目方无法提供可用证书；V1.0 必须仅基于 HTTP 交付可用体验。
- V1.0 不要求把手机文件暴露给浏览器下载，也不要求把手机内容主动推送到浏览器。

## Goals / Non-Goals

**Goals:**
- 提供一个由 React Native 驱动的本地传输服务管理层，统一负责服务启停、地址生成、二维码展示、连接状态和错误提示。
- 提供浏览器访问的响应式投递页，支持文件/文件夹上传、文本粘贴提交和实时进度反馈。
- 为 iOS 与 Android 定义统一的入站文件保存抽象，屏蔽目录选择、权限申请、保存路径与 URI 差异。
- 提供手机端文本接收区，用于展示浏览器提交的文本内容并允许用户复制或清理。
- 提供首版安全控制，包括“简单模式 / 安全模式”切换、连接设备数量限制与服务手动停止能力。
- 服务在 App 切到后台或设备息屏后，仍应保持可访问，并具备被系统干预时的恢复与提示能力。
- 定义真机验证路径，覆盖公共 Wi-Fi 与热点两种网络模式、上传与文本提交成功率和异常恢复。

**Non-Goals:**
- 不在 V1.0 中实现剪贴板同步、传输历史、在线预览等 V2.0 功能。
- 不支持 HTTPS，不提供证书生成、安装、分发或浏览器信任链引导。
- 不提供跨互联网远程传输能力，范围仅限局域网或手机热点直连网络。
- 不要求手机向浏览器发送文件，也不要求浏览器浏览手机文件列表或下载手机文件。

## Decisions

### Decision 1: 采用“React Native 共享控制层 + 轻量原生服务适配层”的分层架构

选择：
- 共享层负责服务配置、状态机、UI、二维码、Web 资产挂载、权限编排和错误展示。
- 原生适配层只处理 React Native 无法稳定覆盖的能力，例如本地 HTTP 服务绑定、平台文件句柄访问、系统权限桥接和网络接口枚举。

原因：
- 这能最大化复用双端代码，同时把 iOS/Android 差异限制在少量适配接口中。
- 纯原生双实现会造成能力漂移和后期维护成本上升。

替代方案：
- 纯 React Native 社区库拼装。问题是本地服务、文件句柄和部分权限行为在双端一致性上风险较大。
- 双端完全原生实现。问题是交付周期长，不符合首版快速成型目标。

### Decision 2: V1.0 以本地 HTTP 服务承载入站投递，不引入浏览器下载或 WebDAV

选择：
- 浏览器访问、文件上传、文本提交和状态查询全部基于 HTTP 路由实现。
- V1.0 不提供手机文件浏览、单文件下载、文件夹打包下载或 WebDAV 暴露能力。

原因：
- 浏览器天然兼容 HTTP，能最快满足“无需安装 App 即可向手机投递内容”的核心价值。
- 下载和 WebDAV 能力会显著增加目录暴露、权限控制与兼容性测试成本，不符合当前单向投递目标。

替代方案：
- 首版同时提供浏览器下载或 WebDAV。优点是功能更完整，但会增加安全面和交付复杂度。

### Decision 3: 浏览器门户页面由 App 内嵌静态资源提供，接口与静态页同源部署

选择：
- 将响应式 Web 页面作为 App 资源内嵌，并由本地服务直接提供 HTML/CSS/JS 与 API。
- 上传和文本提交状态通过同源接口轮询或流式状态返回展示。

原因：
- 这样可以避免额外远端托管，保证离线和局域网内即可工作。
- 同源部署能简化浏览器跨域问题与首版安全策略。

替代方案：
- 使用远端页面托管。问题是不符合“离线局域网可用”的产品价值。

### Decision 4: 文件保存使用统一 Inbound Storage Gateway，按平台映射真实写入能力

选择：
- JS 层定义统一接口：目标目录授权、写入上传内容、处理文件名冲突、删除临时文件、查询保存结果。
- iOS 优先基于应用可访问容器与文档选择器映射接收目录。
- Android 优先基于 Storage Access Framework 与内容 URI 将上传内容写入用户授权目录。

原因：
- 双端文件权限模型差异大，若直接在业务层耦合平台细节，后续所有接收逻辑都难以扩展。
- 统一抽象后，上传落盘流程、错误提示和测试策略都可以共享。

替代方案：
- 直接固定应用沙盒目录。优点是实现简单，但会削弱用户管理接收文件的能力。

### Decision 5: 浏览器提交的文本内容先进入手机端接收区，不直接强制覆盖系统剪贴板

选择：
- 浏览器门户提供文本输入 / 粘贴区，通过 HTTP 接口把文本提交到 App。
- App 侧保存最近一次成功提交的文本内容、来源时间和提交状态，并在接收区提供复制与清空入口。
- 文本提交成功后可以提示用户“一键复制到手机剪贴板”，但不在后台静默覆盖系统剪贴板。

原因：
- 用户需求是“浏览器可以把粘贴内容发到手机”，而不是强依赖底层剪贴板同步协议。
- 直接覆盖系统剪贴板会带来意外覆盖、后台行为不稳定和平台差异问题。

替代方案：
- 浏览器提交后立即写入系统剪贴板。优点是更接近实时粘贴，但会增加平台行为不确定性和误覆盖风险。

### Decision 6: 安全能力采用“简单模式 / 安全模式（URL key）”的 HTTP 策略

选择：
- 提供两种模式：
  - 简单模式：用户只需在浏览器输入 URL（或扫码）即可访问传输门户页。
  - 安全模式：URL 中必须包含一个 `key` 参数（或路径片段）作为访问凭证；缺少或错误的 key 视为未授权。
- `key` 由 App 生成并可一键刷新，刷新后旧 key 立即失效。
- 两种模式都只使用 HTTP，不涉及证书与 HTTPS。

原因：
- 用户侧无法创建 HTTPS，且项目方无法提供证书；将 HTTPS 作为能力会导致不可落地或体验不可控。
- 局域网传输的首要风险是未授权访问；将访问凭证收敛为 URL 中的 `key` 能在不引入证书的情况下提升安全性，并减少额外输入步骤。

替代方案：
- 继续使用“访问密码”弹窗。问题是跨设备输入成本高、容易被忽略，且分享/扫码链路上不如 key 直达。
- 强制 HTTPS。问题是无法满足证书约束，无法交付。
- 完全不做安全控制。问题是风险过高。

### Decision 7: 服务生命周期以“显式启停 + 后台保活”为核心能力

选择：
- 用户进入传输页并授权目录后，显式启动服务。
- App 在前台与后台均保持服务运行，并展示状态、地址、二维码和活动连接数。
- Android 侧以“前台服务 + 常驻通知”的方式保活传输服务，避免后台限制导致服务被挂起。
- iOS 侧以原生后台执行能力（例如后台任务窗口）尽可能维持本地服务可用，并在被系统干预时提供自动恢复/一键恢复能力与清晰提示。

原因：
- 用户核心场景要求“切后台/息屏仍可传”，保活是 V1.0 的必要能力而非增强项。
- Android 与 iOS 的后台策略差异明显，需要在统一体验目标下分别采用平台推荐做法落地。

替代方案：
- 仅前台可用。问题是无法满足产品核心诉求。

## Risks / Trade-offs

- [iOS 局域网权限或本地网络发现受限] → 首次启动服务前集中弹出权限说明，并在失败时提供重试与系统设置引导。
- [Android SAF 目录授权后 URI 行为复杂] → 把目录访问全部收敛到统一网关并在真机上覆盖多品牌机型验证。
- [大文件或整目录上传中断] → 采用流式写入、失败重试和清晰的上传状态提示，必要时分批保存临时产物。
- [目标目录不可写或空间不足] → 在写入前执行可用性检查，并对浏览器端和手机端同时返回失败原因。
- [文本提交内容过大或包含敏感数据] → 限制单次文本长度，默认仅保留最近一次文本并提供用户手动清空入口。
- [HTTP 明文传输在不受信任局域网的泄露风险] → 默认提示“仅建议在可信 Wi-Fi/热点使用”；提供安全模式（URL key）降低未授权访问风险，但不承诺链路加密。
- [热点模式下不同机型 IP 和网卡行为不一致] → 增加网络接口探测与手动刷新地址能力，并在真机验证中覆盖热点场景。
 - [URL key 被截屏/转发导致泄露] → 提供一键刷新 key；可选增加“自动过期”作为后续增强（不作为 V1.0 强制项）。
- [iOS 后台执行窗口与系统调度不确定] → 采用“尽可能持续 + 自动/一键恢复 + 明确状态提示”的组合策略，并将真实机型与系统版本覆盖纳入验收。
- [Android 前台服务带来常驻通知与耗电] → 提供清晰的“停止服务”入口与状态提示，避免用户误以为服务已关闭。

## Migration Plan

1. 初始化 React Native 项目骨架与共享目录结构，预留 `service`, `file-access`, `portal`, `security` 模块边界。
2. 接入本地 HTTP 服务与 Web 静态资源分发能力，先打通地址访问与基础健康检查。
3. 接入入站存储网关与文本接收区，在双端完成目录授权、文件写入和文本落地。
4. 实现浏览器门户的上传、文本提交与进度展示，再补充二维码、简单/安全模式与连接控制。
5. 在 iOS 与 Android 真机上分别验证公共 Wi-Fi、热点、权限拒绝、大文件上传、文本提交和服务中断恢复。
6. 增强后台保活与状态恢复，确保浏览器向手机的主路径在切后台/息屏时仍尽量可用。

回滚策略：
- 若文件落盘能力不稳定，可先关闭目录上传，仅保留单文件上传与文本提交作为缩小范围的临时回退。
- 若安全模式或连接限制引发兼容问题，可关闭对应开关，不影响简单模式主路径。

## Open Questions

- V1.0 是否要求浏览器端支持整文件夹上传，还是允许在不支持目录上传的浏览器中回退为多文件上传。
- iOS 侧首版允许把浏览器上传的文件保存到哪些目录，是仅用户显式授权目录，还是允许映射应用内导入区。
- 浏览器提交的文本内容是否需要“自动写入系统剪贴板”，还是保持“进入接收区后由用户手动复制”的模式。
- 安全模式的 `key` 是否需要默认启用，以及是否需要提供可配置的自动过期时间（例如 10 分钟 / 1 小时）。
