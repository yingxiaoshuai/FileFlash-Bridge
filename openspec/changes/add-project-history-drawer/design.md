## Context

当前工作台在 [App.tsx](d:\company\FileFlash-Bridge\App.tsx) 中将“项目历史”作为常驻左侧区域渲染，桌面宽度下会长期占据一列空间，窄屏下也会把历史列表放在主内容前面。与此同时，项目的创建、切换和删除能力已经在 [useAppModel.ts](d:\company\FileFlash-Bridge\src\app\useAppModel.ts) 与 [inboundStorageGateway.ts](d:\company\FileFlash-Bridge\src\modules\file-access\inboundStorageGateway.ts) 中实现，当前活跃项目仍然是浏览器文件上传、文本提交和共享文件归属的唯一数据入口。

这意味着本次变更的核心不是重做数据层，而是重构项目历史的入口与工作台焦点：把常驻侧栏替换成左上角可展开的历史项目入口，并在用户选中某个历史项目后，稳定回到默认工作台总览态。现有页面没有引入导航库，仍然是单个 `ScrollView` 组合的共享 React Native 页面，因此更适合通过共享 UI 状态与滚动控制完成“回到主页”，而不是新增路由体系。

约束如下：
- 必须继续复用现有 `createProject`、`selectProject`、`deleteProject` 数据能力，不改动持久化结构。
- 浏览器上传文件、浏览器提交文本、共享文件下载和分块重试策略都必须保持现状，仅跟随当前活跃项目变化。
- iOS 与 Android 需要使用同一套 React Native 交互，不新增权限申请，也不引入新的第三方原生库或 Native Module。
- 删除项目时必须保留现有不可忽略的确认提示，避免误删关联文件。

## Goals / Non-Goals

**Goals:**
- 在主工作台左上角提供统一的“项目历史”按钮，支持展开和收起历史项目管理层。
- 在展开层中展示历史项目列表、当前活跃项目选中态、消息数/文件数等摘要，以及创建、切换、删除操作。
- 让用户从历史列表切换项目后，系统关闭展开层并回到默认工作台总览态，形成明确的“切换完成”反馈。
- 保持现有上传文件、文件保存、文本提交、共享文件下载、分块重试和用户提示行为不变，只改变项目切换入口与工作台聚焦方式。
- 在 iOS 与 Android 上复用相同共享实现，并通过真机验证桌面宽度与手机宽度下的展开交互。

**Non-Goals:**
- 不修改项目、消息或文件的持久化模型，不引入新的存储迁移。
- 不让浏览器端直接选择目标项目，浏览器入站内容仍然写入当前活跃项目。
- 不引入 React Navigation、原生侧边栏组件或新的动画/手势库。
- 不改变删除确认文案、上传协议、文本提交协议、共享文件下载协议和分块重试规则。

## Decisions

### Decision 1: 以共享 React Native UI 重构历史入口，不调整存储网关和服务接口

选择：
- 保持 [useAppModel.ts](d:\company\FileFlash-Bridge\src\app\useAppModel.ts) 中的 `createProject`、`selectProject`、`deleteProject` 作为唯一项目管理动作。
- 保持 [inboundStorageGateway.ts](d:\company\FileFlash-Bridge\src\modules\file-access\inboundStorageGateway.ts) 中 `activeProjectId` 的语义不变，浏览器文件上传、文本提交、共享文件仍然依赖当前活跃项目。
- 本次实现只在 [App.tsx](d:\company\FileFlash-Bridge\App.tsx) 增加历史入口、展开层和工作台焦点控制。

原因：
- 现有项目模型已经具备所需能力，继续复用可以避免引入数据兼容风险。
- 需求是用户界面和交互语义升级，而不是协议或数据模型变化。

备选方案：
- 在数据层新增“最近访问项目”或“主页项目”字段。
  不采用，因为现有 `activeProjectId` 已足以表达当前上下文，额外状态会增加一致性负担。

### Decision 2: 用“展开层 + 回到顶部”的方式定义“回到主页”

选择：
- 在 [App.tsx](d:\company\FileFlash-Bridge\App.tsx) 内新增轻量 UI 状态，例如 `isProjectHistoryOpen`。
- 为根 `ScrollView` 增加 ref；当用户从历史项目列表执行“切换项目”或“新建项目”成功后，统一执行：
  - 关闭历史展开层
  - 将新项目设为当前活跃项目
  - 让工作台滚动回顶部，回到默认总览区

原因：
- 当前页面本质上是一个单页工作台，没有真正的路由层。“回到主页”用滚动回顶部最贴合现有结构，也最容易被用户理解为回到默认工作区。
- 这样可以避免为单一交互引入新的导航栈或 tab 状态。

备选方案：
- 引入 `React Navigation`，单独建立主页页和项目详情页。
  不采用，因为改动面过大，且与当前单页式布局不匹配。
- 只切换 `activeProjectId`，不处理工作台聚焦。
  不采用，因为不能满足“选中后回到主页”的明确需求。

### Decision 3: 历史项目使用响应式覆盖层，而不是常驻侧栏

选择：
- 把历史入口放到全局顶部栏左侧，与现有服务状态信息并列。
- 展开层在宽屏下表现为靠左的抽屉式面板，在窄屏下表现为覆盖主内容的全宽面板或底部弹层；两种布局都复用同一份项目列表与操作区。
- 展开层内保留当前项目的高亮态，并在项目卡片上提供切换与删除入口，在底部或头部提供“新建项目”动作。

原因：
- 这能释放主工作台横向空间，让首页总览成为默认视觉中心。
- 在手机和小屏设备上，覆盖层比页面内插入新卡片更容易形成“进入管理层 -> 完成后关闭”的闭环。

备选方案：
- 保留常驻侧栏，只在左上角增加锚点按钮。
  不采用，因为仍然无法解决侧栏长期占位问题。
- 将历史列表改成页面内折叠面板。
  不采用，因为折叠面板展开后仍会推挤主内容，且不利于“管理完成后回到主页”的感知。

### Decision 4: 创建与切换操作关闭展开层，删除操作保留管理上下文

选择：
- “新建项目”沿用现有行为：创建后即成为当前活跃项目；UI 侧额外关闭展开层并回到顶部总览。
- “切换项目”成功后同样关闭展开层并回到顶部总览。
- “删除项目”从展开层发起时继续使用现有确认弹窗。删除成功后刷新列表；若用户仍在历史管理层，则保持展开层打开，便于连续清理历史项目。

原因：
- 创建和切换的目标都是进入新的工作上下文，关闭展开层最自然。
- 删除更偏向管理行为，保留展开层可以减少连续删除或复查时的操作成本。

备选方案：
- 所有操作完成后都强制关闭展开层。
  不采用，因为删除是批量管理场景，强制关闭会打断流程。

### Decision 5: 上传、保存、文本提交和重试链路仅验证“不回归”

选择：
- 本次不改动文件上传接口、文件保存路径、文本提交流程或浏览器分块重试实现。
- 设计文档与测试任务中明确验证：切换项目后，浏览器新提交的文本和文件仍然进入新的活跃项目；共享文件下载能力与分块重试不受影响。
- iOS 与 Android 均不新增权限路径，沿用现有本地网络、文件导出与存储能力。

原因：
- 当前需求只改变手机端项目切换入口，但它会间接影响浏览器写入目标，所以必须做行为验证。
- 不新增权限和原生依赖，可以让双端风险维持在 UI 层。

备选方案：
- 顺手改造浏览器端项目选择能力。
  不采用，因为这会扩展协议范围，超出本次需求。

## Risks / Trade-offs

- [覆盖层与异步项目操作重入冲突] -> 在项目切换、创建、删除进行中复用现有 `busyAction` 禁用重复点击，避免重复触发。
- [“回到主页”在单页布局中语义不清] -> 将其明确为“关闭历史层并滚动回工作台顶部总览区”，并在验收中验证这个行为。
- [窄屏覆盖层可能遮挡删除确认或提示信息] -> 删除动作仍由根级 `Alert` 承担确认，通知条继续显示在主工作台顶部，避免层级冲突。
- [删除当前项目后工作台内容瞬时切换] -> 复用网关现有“自动选择替代项目/自动补一个默认项目”的语义，并在展开层中立刻刷新选中态。
- [桌面宽屏用户失去常驻历史概览] -> 通过顶部入口保留随时展开能力，并在项目卡片中保留消息数、文件数摘要，减轻信息损失。

## Migration Plan

1. 在 [App.tsx](d:\company\FileFlash-Bridge\App.tsx) 中移除常驻历史侧栏的默认展示位置，改为顶部按钮与覆盖层容器。
2. 为根 `ScrollView` 增加 ref，并补充历史层开关、切换后回顶的共享 UI 状态。
3. 将现有项目列表卡片复用于覆盖层，补充创建、删除、切换的入口排布与空态。
4. 验证切换项目后，工作台顶部摘要、项目正文区、浏览器入站目标项目和共享文件归属均更新正确。
5. 若覆盖层方案在某端体验明显退化，可临时回退为页面内折叠面板，但仍保留左上角统一入口与“切换后回顶”行为。

## Open Questions

- 顶部入口文案是否固定为“项目历史”，还是采用图标加短文案的组合；这会影响最终视觉稿，但不影响本次行为设计。
- 删除入口是否直接放在每个项目卡片内，还是先进入选中态后在面板底部统一删除；当前设计优先支持卡片内直接删除。
